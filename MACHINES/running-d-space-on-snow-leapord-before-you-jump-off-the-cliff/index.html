Running D-Space on Snow Leapord, before you jump off the cliff...

2012-04-23
technical notes

yeah, we all like this cute icon.

but getting it to run can be tricky. Then when you pile on PostgreSQL, Maven, Ant, and D-Space 1.8 you may want to run away screaming to jump off the nearest cliff. But don’t jump! It is possible. In this post, you’ll find my notes about how I got D-Space to run on my old Mac laptop. (Of course, you will need the root password so that you can switch back and forth to superuser.) Here are the exact specs of the laptop that we installed on:


Macbook 2.5 GHz Intel Core Duo (4GB ram)
Snow Leapord - OS X 10.6.8

The main documentation is said to be quite comprehensive, but I found it completely mystifying in places, and in the end took some tips from the following sites to get everything running.


Official D-Space Docs
Mac Specific Notes
Configuration Tips

D-Space requires some packages to be installed in advance, so let’s look at them one at a time:


1. JAVA
2. MAVEN
3. ANT
4. POSTGRES
5. TOMCAT

Fortunately, Snow Leopard has a lot of these installed by default! Awesome. Simply fire up your terminal and check your versions. Here are the one’s that were installed on my machine:


$ java -version
java version “1.6.0_31”
$ which javac
/usr/bin/javac
$ mvn -v
Apache Maven 3.0.3
Maven home: /usr/share/maven
$ ant -version
Apache Ant(TM) version 1.8.2

Sweet. We are already running Java 1.6, Maven 3.0.3, and Ant 1.8.2. Now we have to install PostgreSQL and Tomcat. Well, PostgreSQL is going to take you a while, if you have never installed it before. You can follow the instructions here, that I prepared for PostgreSQL and PostGIS, or you can use MacPorts, Fink, or Enterprise DB methods.

Perhaps the trickiest part of the install, for me, was to get all of the users and owners of the libraries configured correctly. In the end what worked for me was to install PostgreSQL as root, then to create a new Mac user account called dspace. After creating the dspace account, I logged out of my default user and logged into the dspace user account. After logging in as the new user “dspace” when you open a terminal shell, you should see something like this:


Last login: Sun Apr 22 12:52:50 on ttys001
yourComputerName:~ dspace$

This will make things a whole lot easier, in the long run! Now that you have PostgreSQL installed you should be able to check the version in your terminal:


dspace$ pg_config - version
PostgreSQL 9.1.1

Note, in the command lines, the prefix will indicate whether you should be running the command as dspace$ or as root#. First, let’s sudo to root and create a new user in postgreSQL. Then drop back to your dspace user to create a database called “dspace.”


dspace$ su -
Password:
root# createuser - superuser dspace -U postgres
root# exit
dspace$ createdb dspace

Note: originally, I tried to create a database with a different name, but that failed on the ANT install. So it appears that the first instance should definitely be called “dspace.” You should now be able to connect to psql and check out the new database.


dspace$ psql -h localhost -U dspace -d dspace
psql (9.1.1)
dspace=# \l
List of databases
Name | Owner | Encoding | Collate | Ctype | Access privileges
————— - +———-+———-+———+——-+——————— -
dspace | dspace | UTF8 | en_US | en_US |
dspace=# \q

If this works, then you’ve succeeded in creating both the new psql user called dspace and the new database called dspace. I wished that I could have used some other user and database names to be less confusing, but unfortunately, this was the ONLY way that worked. But the good news, is that we can now move on to Tomcat. First, you want to download the latest library, in my case, version apache-tomcat-7.0.27. Let’s assume that your download location is called: /Users/X/Downloads. Then we need to create a folder for the package with your user dspace as the owner! Finally, we can unpack the Tomcat source to the new Tomcat folder.


dspace$ su -
Password:
root# cd /Library
root# mkdir Tomcat
root# cd Users/X/Downloads/
root# tar -xzvf apache-tomcat-7.0.27.tar.gz
root# cd /Users/X/Downloads/apache-tomcat-7.0.27
root# cp -fr * /Library/Tomcat
root$ chown -R dspace /Library/Tomcat

Okay, now you should have the Tomcat library owned by your dspace user. To make it operational, it will be necessary to edit the configuration file, called tomcat-users.xml.

By default you will find a section that looks like the following, and which is commented out.



To this should be added some basic roles, including manager-gui, which is not enabled by default. Also, the comment tags should be removed. You can use your favorite editor to do this. *disclosure* I use vi (not emacs)…


dspace$ cd /Library/Tomcat/conf
dspace$ vi tomcat-users.xml

[esc, shift+ZZ]
dspace$

Now that you have installed Tomcat, it is a good idea to make sure that the dspace user has PATH environment variables for both Java and Tomcat. You can get some tips on that here. Meanwhile, the following should work.


dspace$ cd /Users/dspace
dspace$ vi .profile
[add these lines]
JAVA_HOME=/Library/Java/Home
export JAVA_HOME;
CATALINA_HOME=/Library/Tomcat
export CATALINA_HOME;
[esc, shift+ZZ]

At this point, you should close the terminal shell and open a new one to make sure the .profile changes were effective. In the new terminal shell check to make sure the the paths to Java and Tomcat are in the dspace user environment:


dspace$ echo $JAVA_HOME
/Library/Java/Home
dspace$ echo $CATALINA_HOME
/Library/Tomcat

Okay, now to test Tomcat. You want to go to the bin folder and run the startup and shutdown commands. If those work, it is a good thing! Just for convenience, I downloaded the Tomcat Controller and placed it in my apps. You can set the preferences to your Tomcat folder, and it makes it easy to start, stop, and jump to port 8080. Note: it’s important the Tomcat runs under the dspace user environment.


dspace$ cd /Library/Tomcat/bin
dspace# ./startup.sh && tail -f ../logs/catalina.out

You should get some messages and the screen will appear to hang without returning to the dspace$ prompt. At this point, open a browser and go to http://localhost:8080/. If you see the Apache Tomcat welcome screen, all is well! Now just make sure you can shut down Tomcat, but going back to your terminal and get your prompt back by typing CONTROL+C.


dspace# ./shutdown.sh && tail -f ../logs/catalina.out

So far, so good! If you’ve made it this far, then Java, Maven, Ant, Postgres, & Tomcat are all ready to go! We are moving on to the actual installation of D-Space, and first you can download the latest build, which in my case was dspace-1.8.2-src-release.tar.gz. After you download the source, you will want to create a folder where the project will live with a name like “dsource”, then make sure that the owner of the folders (recursively) is the dspace user.


dspace# cd /Users/dspace/Downloads
dspace# mv dspace-1.8.2-src-release.tar.gz /Users/dspace
dspace# cd /Users/dspace
dspace# tar -zxvf dspace-1.8.2-src-release.tar.gz
dspace# su -
root# cd /usr/local
root# mkdir dsource
root# chown -R dspace dsource
root# exit
dspace# cd /Users/dspace/dspace-1.8.2-src-release
dspace# cp -fr * /usr/local/dsource

Once that is done, we go to the configuration step. This part was quite confusing to me, for some reason. In my case, I edited the following lines in the dspace.cfg file to get the ANT install process to work.


dspace# cd /usr/local/dsrc/dspace/config
dspace# cp dspace.cfg dspace_orig.cfg
dspace# vi dspace.cfg
dspace.dir = /usr/local/dsource/dspace
dspace.hostname = localhost
dspace.url = http://localhost:8080/dspace
dspace.name = DSpace on Vishanti
db.name = postgres
db.url = jdbc:postgresql://localhost:5432/dspace
db.driver = org.postgresql.Driver
db.username = dspace
db.password = YourDspaceUserPWD
[esc, shift+ZZ]

At this point the configurations should all be okay, and you can move on to the MAVEN package build and ANT installations. When I ran Maven it took 11 minutes to finish the build. Be prepared to watch a million lines of verbose installation messages when you run Maven! Ok, to get started, change directories to the dspace package folder, then run Maven.


dspace# cd /usr/local/dsource/dspace
dspace# mvn package

Maven will complete with a bunch of INFO lines summarizes the success or failure. You might want to copy these for future reference.


[[INFO] Copying files to /usr/local/dsource/dspace/target/dspace-1.8.2-build]
[INFO] ————————————————————————

[INFO]

According to the common problems FAQ, the most common error is to get the database connection wrong in the dspace.cfg file. So, before running ANT, it’s a good idea to double check that you can connect to your intended pqsl database instance, called “dspace.” Then go ahead and run ANT!


dspace# psql -U dspace -d dspace -h localhost
psql (9.1.1)
dspace=# \q
dspace$ cd /usr/local/dsource/dspace/target/dspace-1.8.2-build
dspace$ ant fresh_install

Ant is pretty quick, compare to the Maven build. In a matter of seconds you should know the outcome. If there are configuration errors, you should get some clue as to what went wrong. And if it worked, you will get some very useful lines about the build, which you might want to copy and save.


[echo] ====================================================================
[echo] The DSpace code has been installed, and the database initialized.
[echo]
[echo] To complete installation, you should do the following:
[echo]
[echo] Setup your Web servlet container (e.g. Tomcat) to look for your
[echo] DSpace web applications in: /usr/local/dsource/dspace/webapps/
[echo]
[echo] OR, copy any web applications from /usr/local/dsource/dspace/webapps/ to
[echo] the appropriate place for your servlet container.
[echo] (e.g. ‘$CATALINA_HOME/webapps’ for Tomcat)
[echo]
[echo] Make an initial administrator account (an e-person) in DSpace:
[echo]
[echo] /usr/local/dsource/dspace/bin/dspace create-administrator
[echo]
[echo] * Start up your servlet container (Tomcat etc.)
[echo]
[echo] You should then be able to access your DSpace’s ‘home page’:
[echo]
[echo] http://localhost:8080/dspace
[echo]
[echo] You should also be able to access the administrator UI:
[echo]
[echo] http://localhost:8080/dspace/dspace-admin

Looks good! Of course, there are some more items to clean up, not the least of which is that the webapp servlets created in the dspace build must be copied to tomcat, and a dspace administrator must be set up.


dspace# cp -R /usr/local/dsource/dspace/webapps/* /Library/Tomcat/webapps
dspace$ /usr/local/dsource/dspace/bin/dspace create-administrator
Creating an initial administrator account
E-mail address: mickey@yankees.com
First name: Mickey
Last name: Mantle
WARNING: Password will appear on-screen.
Password: theMick
Again to confirm: theMick
Is the above data correct? (y or n): y
Administrator account created

The proof is in the pudding! We can now fire up Tomcat and see what is going on.


dspace$ cd /Library/Tomcat/bin
dspace# ./startup.sh && tail -f ../logs/catalina.out

Go to your browser empty the cache. Quit the browser and relaunch it just for good measure. Now let’s load up the page http://localhost:8080/ again. This time, click on the “Manager App” link, and you will be prompted for the username and password which were entered in the tomcat-users.xml file. In my example, above, I used: username=”dspace” password=”AnyPWDYouWant”. The Manager App should accept the credentials that you set in the tomcat-users.xml configuration, and you will be able to view all your current webapps. Because we copied all the dspace webapps into the tomcat webapps folder, those will now be listed. If you scroll down to the bottom, you will find the default User Interface for D-Space, called /xmlui. Click on this link and you should be viewing the default home page of the Manakin UI. You will see a LOGIN link on the upper left corner of the Manakin page. Click that and you will be able to enter the credentials of your “initial administrator account.” In my example, these would be: E-mail address: mickey@yankees.com Password: theMick

Okay, you should now have entered the D-Space management console, and you are up and running!

Congratulations, amigos!
